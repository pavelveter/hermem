## 1. Git Flow

Работаем по **GitHub Flow**:

```
main  ← feature/xxx  ← (работа здесь)
  ↓
PR → review → merge → deploy
```

### 1.1 Ветки

- **`main`** — всегда стабильная, готовая к деплою. В неё ничего не пушат напрямую.
- **`feature/<название>`** — каждая новая фича или изменение. Ответвляется от `main`.
- **`fix/<название>`** — для багфиксов.
- **`refactor/<название>`** — для рефакторинга без изменения поведения.

### 1.2 Коммиты

- **Частые** — коммитить каждое логически завершённое изменение.
- **Осмысленные сообщения** — на русском или английском:
  - Плохо: `fix`, `update`, `changes`
  - Хорошо: `Добавлен гибридный поиск (vector + FTS5) с RRF`, `fix: обработка пустого фида при старте`
- **Префиксы** (опционально, Conventional Commits):
  - `feat:` — новая фича
  - `fix:` — исправление бага
  - `refactor:` — рефакторинг
  - `test:` — тесты
  - `docs:` — документация
  - `chore:` — инфраструктура, CI/CD, зависимости

### 1.3 Pull Requests

- Каждая ветка → отдельный PR в `main`
- PR должен быть небольшим (до ~300 строк изменений)
- PR должен проходить CI (линтеры, type check, тесты) перед мержем
- После мержа ветка удаляется

### 1.4 Pre-push hook

Перед каждым `git push` автоматически запускается `.githooks/pre-push` — все проверки должны пройти, иначе пуш блокируется.

**Включение хука** (один раз на клоне):
```bash
git config core.hooksPath .githooks
```

**Что проверяется** (в порядке запуска):
1. **Architecture guardrails** — поиск запрещённых паттернов (`ActiveSchema()`, экспортированный мутабельный стейт)
2. **gofmt** — форматирование должно быть чистым
3. **go vet** — статический анализ `./src/...`
4. **go build** — сборка `./src/...`
5. **Unit-тесты** — `go test -race -count=1 -timeout 10m ./src/...`
6. **E2E-тесты** — `go test -p 1 -timeout 5m ./tests/e2e/...` (CLI + HTTP + scenarios)
7. **golangci-lint** — линтер (если установлен локально)
8. **AMX guard** — проверка AMX-инструкций (только Darwin + CGO)

**Обход** (только в исключительных случаях): `git push --no-verify`

## 2. Процесс разработки

### 2.1 Шаг 1: Начать новую фичу

```bash
git checkout main
git pull origin main
git checkout -b feature/<название>
```

### 2.2 Шаг 2: Работа

- Менять только то, что относится к фиче
- Документировать публичные API (docstrings)
- Писать тесты на новую логику
- **Коммитить на каждой итерации**: после каждого логически завершённого блока изменений
  (фича, фикс, рефакторинг) — сразу закоммитить с осмысленным сообщением.
  Не копить изменения до PR. Это даёт:
  - Чёткую историю — легко откатить один фикс, не теряя остальные
  - Быстрый review — каждый коммит маленький и понятный
  - Безопасность — изменения зафиксированы, даже если следующие сломают код
